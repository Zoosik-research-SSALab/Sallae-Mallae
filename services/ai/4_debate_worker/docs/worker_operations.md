# 워커 운영 및 복구 가이드

## 운영 목표

- 실행만 하면 계속 작업이 돌아갈 것
- 중간 장애가 나도 작업 내역이 날아가지 않을 것
- 비싼 로컬 LLM 추론 결과는 가능한 재사용할 것

## 현재 동작 방식

1. `debate-targets`로 대상 종목 조회
2. 체크포인트 DB에 대상 동기화
3. 처리 가능한 다음 작업 1건 claim
4. 입력 조회
5. 토론 실행
6. 결과 payload를 체크포인트 DB에 먼저 저장
7. 결과 API POST
8. 성공 또는 중복이면 완료 처리
9. 실패 시 재시도 또는 영구 실패 처리

## 체크포인트 상태

- `pending`
  - 아직 처리하지 않은 상태
- `running`
  - 현재 작업 중이거나 비정상 종료로 중간에 멈춘 상태
- `result_ready`
  - 토론 결과는 생성됐고 결과 저장 API 재시도만 필요한 상태
- `succeeded`
  - 결과 저장 완료
- `duplicated`
  - 서버에 이미 같은 결과가 있어 신규 저장은 생략
- `failed_retryable`
  - 재시도 가능한 실패
- `failed_permanent`
  - 재시도하지 않는 실패

## 장애별 대응

### 1. 데스크탑 전원 종료

증상:
- 작업 중간에 프로세스 종료

복구:
- 워커 재실행
- lease 만료된 `running` 작업부터 자동 복구
- `result_ready`는 재추론 없이 결과 POST부터 재개

### 2. EC2 API 일시 장애

증상:
- 5xx, 타임아웃, 연결 실패

복구:
- `failed_retryable`로 저장
- backoff 후 자동 재시도

### 3. 로컬 LLM 응답 형식 오류

증상:
- JSON 파싱 실패

복구:
- retryable 실패 처리
- 반복 실패 시 모델 프롬프트/모델 품질 점검

### 4. 잘못된 API 키

증상:
- 401 반환

복구:
- `.env`의 `INTERNAL_API_KEY` 수정
- 이 경우는 보통 영구 실패 원인이므로 키를 먼저 고친 후 재실행

## 운영 팁

- 대상은 초기에는 `trading_history` 기반으로 유지
- 결과 로그가 길수록 느려지므로 프롬프트는 짧게 유지
- 모델을 바꾸면 `LLM_MODEL`과 `WORKER_NAME`을 함께 관리
- 여러 대를 돌릴 때는 각 PC에 고유한 `WORKER_NAME`을 사용

## 체크포인트 초기화

특정 실험을 처음부터 다시 하고 싶을 때만 `state/worker_checkpoints.sqlite3`를 백업 후 삭제합니다.

주의:
- 이 파일을 지우면 로컬 복구 정보가 모두 사라집니다.
- 이미 서버에 저장된 결과는 다시 `duplicated` 처리될 수 있습니다.

## 추천 운영 시나리오

### 시나리오 A: 하루 배치

- 매일 저녁 또는 새벽 1회 실행
- `python main.py --once --report-date YYYY-MM-DD`

### 시나리오 B: 계속 실행

- 데스크탑이 켜져 있는 동안 계속 polling
- `python main.py`
- 주기적으로 새 대상이 생기면 자동 처리

개인 데스크탑 운영은 시나리오 B가 가장 편합니다.
